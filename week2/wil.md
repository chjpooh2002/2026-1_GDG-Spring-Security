# 🛡️ Spring Security: 세션 기반 인증 방식 (Session-based Authentication)

스프링 시큐리티의 핵심인 **세션 인증 메커니즘**과 **SecurityFilterChain** 빈 등록 원리에 대해 정리한다.

---

## 1. SecurityFilterChain 빈 등록 원리
스프링 시큐리티의 모든 설정은 `SecurityFilterChain` 빈을 등록하는 것에서 시작한다. 개발자가 설정을 직접 구현하지 않으면 스프링 부트는 특정 조건에 따라 **DefaultSecurityChain**을 빈으로 자동 등록한다.

### 🔍 자동 생성 조건 (Conditional)
1. **의존성 존재:** 클래스패스에 시큐리티 필터 체인과 `HttpSecurity` 관련 라이브러리가 존재할 때 생성된다.
2. **빈 부재 (Default):** 개발자가 직접 등록한 `SecurityFilterChain` 타입의 **빈(Bean)이 없을 때**만 생성된다.

> **💡 핵심 포인트**
> 개발자가 직접 빈을 구현하여 등록하면 기본 스프링 시큐리티 빈은 비활성화된다. 기본 설정은 사용자 1명(`user`)만 처리할 수 있으므로, 실무에서는 반드시 직접 빈을 구현해야 한다.

---

## 2. 세션 인증의 핵심: Stateful
스프링 시큐리티의 기본 인증 방식은 **세션(Session)**을 기반으로 하며, 상태를 유지하는 **Stateful**한 특성을 가진다.

* **JSESSIONID**: 세션 인증의 핵심 키값이며 클라이언트 브라우저의 쿠키에 저장된다.
* **상태 유지**: 서버가 클라이언트의 인증 상태를 관리하는 방식이다.

---

## 3. 빈(Bean)의 성격과 스코프
* **HttpSecurity (Prototype)**: 매 주입마다 새로운 객체를 생성하기 위해 **프로토타입**으로 정의되어 있다. 이는 각 설정 간의 독립성을 보장하기 위함이다.
* **일반 설정 빈 (Singleton)**: 어디서든 사용되므로 **상태를 가져서는 안 된다.**

---

## 4. 로그인 프로세스 (Login Workflow)

로그인 요청 시 내부적으로 다음과 같은 5단계 과정을 거쳐 인증이 완료된다.

| 순서 | 단계 | 상세 내용 |
| :---: | :--- | :--- |
| **1** | **객체 변환** | 로그인 요청 정보를 `Authentication` 객체(authToken)로 변환한다. |
| **2** | **인증 시도** | `AuthenticationManager`가 해당 객체로 인증을 시도한다. (실패 시 `AuthenticationException` 발생) |
| **3** | **정보 저장** | 인증 성공 시, 현재 인증 정보를 `SecurityContext`가 담당하여 저장한다. |
| **4** | **세션 발급** | 서버 세션을 생성하고 클라이언트에 `JSESSIONID`를 발급한다. |
| **5** | **최종 저장** | 세션에 인증 정보를 저장하여 이후 요청에서도 인증 상태를 유지한다. |

---

## ⚠️ 주의사항
* **초기 계정**: 기본 설정 시 ID는 `user`이며, PW는 애플리케이션 실행 시 **콘솔창**에 출력된다.
* **빈 구현 필수**: 기본 설정 그대로는 다중 사용자를 처리하기 어렵다. 따라서 직접 스프링 시큐리티 빈을 구현하여 디폴트 빈을 대체해야 한다.

##MemberRole의 value로 ROLE 접두사를 붙이는 이유
스프링 시큐리티는 "권한(ROLE)"을 처리할 때 "ROLE_" prefix가 붙은 문자열을 권한으로 인식한다.
hasRole("REGULAR")
→ 내부적으로 "ROLE_REGULAR"로 변환됨
따라서 DB / Enum / 권한 값은
👉 반드시 "ROLE_" prefix 포함해야 함
